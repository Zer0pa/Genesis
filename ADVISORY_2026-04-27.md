## §1. From-scratch design (clean slate, what I'd build)

Three layers, leveraging the fact that Genesis source is **open** (unlike dm3_runner closed binary):

### §1.1 Determinism layer (Rust, leveraging open source)

A `genesis_runner` binary built from `genesis_cli` (existing CLI entry point) plus a new `genesis_harness` crate that:
- Implements the dm3 task surface (`--task exp_k2_scars --steps N`) on the Genesis substrate
- Emits dm3-mirror JSON receipts per invocation: `canonical_sha`, `receipt_sha`, `env_pre`, `env_post`, `task`, `--steps`, `best_uplift`, `max_scar_weight`, `timestamp_utc`, plus `genesis_meta` sub-object (for build params: `turns`, `verify_hash`, `solve_hash`, source git commit, target triple, build flags)
- **Higher-precision option** (`--receipt-format=json-bigrat`): emits `BigRational` form bypassing the printf-6-decimal floor. This is the determinism enhancement Rust enables that dm3_runner's closed binary doesn't expose — it solves OAI-2 (precision floor) on Genesis side natively.
- **Internal-state checksum at every `--steps` boundary**: emit a per-step state hash, not just the final scalar. Enables structural-deterministic verification, not just output-determinism verification.

### §1.2 Speed layer (multi-process across cores)

Multi-*process* via shell orchestration on multiple cores (mirrors dm3_parallel pattern, ports it to genesis_runner):
- 6 parallel `genesis_runner` invocations, each taskset-pinned to one of cpu0..cpu5
- cpu6 = thermal margin (scale to cpu0..6 = 7 cores if max thermal stays under 75°C across first long cell)
- cpu7 = untouched (dm3_runner's)
- Per-cell completion → spin up next batch
- This is "multithreading" in the OS-process sense, not Rust-internal-rayon sense (the latter is unnecessary here; tasks are inter-invocation parallel)

For specific cells where intra-invocation parallelism would help (e.g., spectral analysis at high precision, parameter Monte Carlo), genesis_harness can use `rayon` (already declared in workspace deps). But this is optional and per-cell decided.

### §1.3 Continuity layer (resume-safe chain)

Master orchestrator + watcher + per-cell state files (mirrors dm3 phase_g_chain pattern):
- `genesis_chain_v1.sh` reads `cells/<CELL>/_state.json`, skips completed cells, executes pending
- `master_watcher.sh` polls master pid; restarts via `resume_chain.sh` if dead
- Per-cell completion writes `{cell, verdict, summary, metrics, next_actions}.json`
- Chain log appended atomically
- `pidof dm3_runner` collision check before any device probe

### §1.4 Test program (mirrors PRD §3 cells)

Phase 2 (full PRD execution, after Phase 1 K2 port lands):
- **BITDET**: bit-determinism — N=10 at fixed config, all canonical SHAs match. ~30 min.
- **PARITY**: cross-platform M1↔RM10 parity — same fixed config, compare canonical SHAs. ~10 min.
- **SYMMETRY**: D₆ test (Z₂ mirror axis specifically). Pick observable sensitive to mirror-asymmetric mode. ~30 min.
- **K2_SWEEP**: K2 protocol on Genesis, `--steps` ∈ {20, 28..56}, N=3 per step → 90 invocations × ~30s ÷ 6 cores = ~7-8 min wall.
- **CYCLE**: cycle probe — multiples of 7, 6, 8 disambiguation. ~3-4 min.
- **CROSSBUILD**: vary `turns` knob (default 4 → P_95 ⋉_α Z_3 substrate); K2 sweep on at least one variant. ~10-15 min per variant.
- **DISCONT**: only if cliff observed in K2_SWEEP; else record "no cliff observed in [range]". ~10 min.

### §1.5 The four pre-registered comparison verdicts (PRD-aligned)

1. **Cycle-7 attribution**: does Genesis cycle at period 7 (substrate-attributed via T(3,21)'s seven full twists), or at a different period (augmentation-attributed)?
